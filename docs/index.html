Built for Modern Data Loads

LumeDB abandons the traditional B-Tree for an LSM-Tree architecture, providing massive write throughput without sacrificing read latency.

LSM-Tree Storage

Append-only MemTable writes flushed to immutable SSTables. Eliminates the write-amplification bottleneck of traditional databases.

ACID Transactions

Full MVCC implementation with snapshot isolation. Transactions commit atomically via the Write-Ahead Log (WAL).

Bloom Filters

Every SSTable includes an embedded bloom filter, allowing O(1) skipping of disk reads for non-existent keys.

LZ4 Compression

All SSTable data blocks are heavily compressed, significantly reducing storage footprint and disk I/O.

Vector Search

Built-in cosine similarity, euclidean distance, and dot product search. Store embeddings directly alongside your documents.

Filtered Vector Queries

Combine metadata filters with vector search in a single query. Pre-filter by category, date, or any field before similarity ranking.

Simple. Familiar. Powerful.

LumeDB uses a familiar JSON-based query DSL over a standard TCP connection. Use the interactive CLI or build your own driver.

  • Native JSON document storage
  • Full support for $eq, $gt, $in, $or
  • Update operators like $set, $inc, $push
  • Fast secondary B-Tree indexes
// 1. Insert a document
lume> db.users.insert({
  "name": "Alice",
  "age": 30,
  "skills": ["rust", "python"]
})
{"status":"ok", "insertedId":"65926216-..."}

// 2. Query with operators
lume> db.users.find({
  "age": {"$gte": 25},
  "skills": {"$in": ["rust"]}
})
{"status":"ok", "count": 1, "documents": [...]}

// 3. Update document
lume> db.users.update(
  {"name": "Alice"}, 
  {"$inc": {"age": 1}}
)
{"status":"ok", "modifiedCount":1}

One Database for RAG

Stop syncing data between your document store and a separate vector database. LumeDB stores your content, metadata, and embeddings together.

  • Cosine, Euclidean, and Dot Product metrics
  • Automatic embedding detection on insert
  • Pre-filter by metadata, then rank by similarity
  • No Pinecone, no Qdrant — one standalone binary

RAG Pipeline with LumeDB

  1. Chunk your PDF or text
  2. Generate embeddings via OpenAI or a local model
  3. Store chunk + metadata + embedding in one document
  4. Query with vectorSearch to retrieve relevant context
// 1. Create a vector index
lume> {
  "action": "createVectorIndex",
  "collection": "articles",
  "field": "embedding",
  "dimensions": 1536,
  "metric": "cosine"
}

// 2. Insert doc with embedding
lume> {
  "action": "insert",
  "collection": "articles",
  "document": {
    "title": "Intro to Rust",
    "embedding": [0.9, 0.1, ...]
  }
}

// 3. Similarity search + filter
lume> {
  "action": "vectorSearch",
  "collection": "articles",
  "vector": [0.85, 0.2, ...],
  "k": 5,
  "filter": {"category": "programming"}
}
→ score: 0.993, "Intro to Rust"
